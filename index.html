Revision 2.0 — Lythos Kernel

Capability-Secure
Microkernel OS

RaptorOS is built on the Lythos microkernel — a minimalist, Rust-native kernel where all drivers and services run in userspace. Capability tokens, atomic rollbacks, and a content-addressed store provide a foundation that is fault-isolated by design.

Core Architecture

Lythos exposes a minimal syscall surface. All drivers and services run as isolated userspace processes — a crash anywhere outside the critical triad cannot affect the kernel.

Kernel
Lythos Microkernel
Minimalist and high-performance. Syscalls are grouped into four categories: memory management, IPC primitives, capability operations, and scheduling. Nothing more.
// Syscall surface
memory | ipc | cap | sched
Security
Capability Model
Unforgeable tokens granted by lythdist at boot. No ambient authority — every hardware access and IPC region requires a token. Tokens cannot be constructed or duplicated in userspace.
IPC
Shared Memory Bus
All inter-process communication flows through lythmsg. Transport is shared memory; discovery is token-based. Allocated by lythdist, async by design.
lythdist → allocates region
lythmsg  → async message passing
Language
Rust Throughout
Memory safety enforced at the kernel level. Userspace toolchain targets x86_64-unknown-linux-musl with full std support via statically-linked musl libc.

Boot Sequence

Strictly ordered. lythmsg does not become available until lythdist has distributed capabilities. Non-critical services do not start until the core is confirmed healthy.

01
Lythos kernel
Initializes, mounts /lth/system as immutable read-only
02
lythd
Starts as PID 1
03
lythd → lythdist
Spawns lythdist; reads hardware topology and allocates capability tokens
04
lythdist → lythd
Grants lythd its capabilities, opens for service requests
05
lythd → lythmsg
Spawns lythmsg using granted shared memory region
06
lythd
Checks for pending rollback flag from rpkg. If present, starts 30-second stability timer
07
lythd
Mounts remaining subvolumes: /lth/store, /cfg, /user
08
lythd
Reads service definitions, spawns non-critical services in dependency order
09
lythd [rollback path]
If timer active and critical daemon fails: atomically reverts /lth/system + /cfg snapshots, reboots
10
lythd
Timer expires cleanly: marks snapshot stable, clears rollback flag
11
System live
User sessions spawn lysh on demand
03

Filesystem Layout

Btrfs subvolumes with clear ownership boundaries. The /lth/ namespace is system-owned. Symlinks in /lth/bin point to active versions in the content-addressed store.

Path Subvolume Writable Description
/lth/system @core No Immutable. Lythos kernel and lythd.
/lth/store @store No* Content-addressed binary store. Never mutated in place. rpkg mounts rw transiently during installs.
/lth/bin N/A No rpkg-managed symlinks pointing to active versions in /lth/store.
/cfg @cfg Yes System configuration. Snapshotted atomically with /lth/system before every update.
/user @home Yes Persistent user data. Not included in system rollbacks.

Tooling Stack

lyth- prefix for kernel-adjacent daemons. rp- prefix for user-facing OS tools. Clean boundary, no ambiguity.

lyth- / PID 1
lythd
Init process and service supervisor. Manages boot sequence, stability timer, and rollback coordination. The root of the process tree.
Critical PID 1
lyth- / capabilities
lythdist
Hardware capability distributor. Reads topology at boot, allocates unforgeable tokens, grants access to IPC memory regions.
Critical
lyth- / IPC
lythmsg
Async IPC bus. Shared-memory transport. Token-based discovery via lythdist. A crash triggers an immediate system rollback.
Critical IPC Bus
rp- / packages
rpkg
Source-first package manager wrapping Cargo. Atomic installs, binary caching via Btrfs reflinks, config snapshots, and rollback flag management.
Source-First
rp- / observability
rpview
Live TUI showing the service dependency graph and lythmsg bus health in real time. Read-only — observe without interference.
rp- / chaos
rpbreak
Chaos engineering tool. Manually kills services to test supervision, restart behavior, and rollback triggers under controlled conditions.
shell
lysh
Native system shell. POSIX-compliant. Launched as a user session process on demand — not part of the boot sequence, not in the critical set.
05

Package Manager

rpkg is source-first and always builds via Cargo. The binary cache skips recompilation on content-hash match. Btrfs reflinks deduplicate identical outputs without copying.

1
Snapshot
Atomically snapshot /lth/system and /cfg as a pair before any change
2
Set Flag
Write rollback flag — activates 30-second stability timer on next boot
3
Resolve Deps
Cargo resolves native deps. C dependencies without a -sys crate are built from the [system-deps] block, installed to /lth/store, and injected into Cargo's environment
4
Cache Check
Check /lth/store for a matching content hash. On hit: reflink — no recompilation
5
Build
On cache miss: build via Cargo, install result to content-addressed store
6
Activate
Update /lth/bin symlinks atomically to point to new store entries
7
Confirm
On clean boot past stability window: mark snapshot stable, clear rollback flag
06

Resilience

Fault isolation by architecture. Automatic rollback by policy. Config never diverges from system state.

⊗
Fault Isolation
All drivers and non-critical services run as isolated userspace processes. A crash in lynet, lygpu, or any non-critical service does not reach the kernel or critical daemons. lythd detects via its supervision loop and restarts the service automatically.
↩
Automatic Rollback
Every rpkg update sets a rollback flag. On the next boot, lythd starts a 30-second stability timer. Any critical daemon failure within that window triggers an atomic Btrfs rollback of /lth/system and /cfg together — they are never out of sync.
