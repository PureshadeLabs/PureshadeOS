§1 Lythos Microkernel
Architecture

Lythos is a minimalist, high-performance microkernel. The design principle is maximal isolation with minimal privilege surface. All device drivers and OS services run as ordinary userspace processes — the kernel itself provides only the primitives required to build everything else.

Security ModelCapability-based — unforgeable tokens grant access to hardware and IPC regions
LanguageRust — memory safety enforced at the kernel level
IPC PrimitiveShared memory regions with async message passing, allocated by lythdist

Lythos exposes the minimum syscall surface required for a complete OS. All syscalls fall into exactly four categories.

Note
Capability tokens are required for all memory mapping and IPC operations. There is no ambient authority — processes cannot access hardware or communicate without a token granted by lythdist.
IPC Model

All inter-process communication on RaptorOS flows through lythmsg, which is built on top of lythdist-allocated shared memory regions. The transport layer is always shared memory — no kernel-mediated message copying occurs after the region is established.

// IPC region lifecycle
lythdist allocates shared memory region → grants capability token to participants
lythmsg  establishes async channel over region
producer writes to region → ipc_send(region_cap, msg)
consumer polls or blocks → ipc_recv(region_cap) → msg
// No kernel copies after region establishment
Core Daemons

Three daemons form the critical set. A failure in any of them during the 30-second stability window triggers an automatic system rollback.

Critical Set
lythd, lythdist, and lythmsg are the only processes whose failure triggers a rollback. All other services are non-critical and are restarted by lythd on crash.
lythd
PID 1 · Init · Supervisor
Supervises all services, manages the boot sequence and stability timer, and coordinates rollbacks on critical failure. Root of the process tree.
lythdist
Capability Distributor
Reads hardware topology at boot, allocates unforgeable capability tokens, and grants access to shared memory regions used by lythmsg.
lythmsg
IPC Bus
Async shared-memory message passing for all inter-process communication. Built on lythdist-granted memory regions. Token-based service discovery.
§2 Boot Sequence
Step Order

The boot sequence is strictly ordered. lythmsg does not become available until lythdist has distributed capabilities. Non-critical services do not start until the core is confirmed healthy.

#ActorAction
1Lythos kernelInitializes, mounts /lth/system (immutable, read-only)
2lythdStarts as PID 1
3lythd → lythdistlythd spawns lythdist; reads hardware topology and allocates capability tokens
4lythdist → lythdGrants lythd its capabilities and opens for service requests
5lythd → lythmsgSpawns lythmsg using granted shared memory region
6lythdChecks for pending rollback flag from rpkg. If present, starts 30-second stability timer
7lythdMounts remaining subvolumes: /lth/store, /cfg, /user
8lythdReads service definitions, spawns non-critical services in dependency order
9lythdRollback path: if timer active and critical daemon fails — atomically reverts /lth/system + /cfg, reboots
10lythdTimer expires cleanly: marks snapshot stable, clears rollback flag
11—System is live. User sessions spawn lysh on demand
Stability Timer

The stability timer is activated only when a rollback flag is present — i.e., on the first boot after an rpkg install or update. Normal boots without a pending rollback flag proceed directly to step 7 without a timer.

Timer Behavior
Duration is 30 seconds from the point the flag is detected (step 6). The timer covers the entire period through which non-critical services are started. If a critical daemon fails at any point before expiry, rollback is triggered regardless of which step the boot was on.
§3 Filesystem Layout
Subvolume Layout

RaptorOS uses a Btrfs subvolume-based structure. The /lth/ namespace is system-owned and managed exclusively by rpkg and the kernel. Symlinks in /lth/bin are managed by rpkg and point to active versions in /lth/store.

PathSubvolumeWritableDescription
/lth/system @core No Immutable. Lythos kernel and lythd.
/lth/store @store No* Read-only store for compiled binaries. Content-addressed, never mutated in place. Functions like /nix/store. (*rpkg mounts rw transiently during installs.)
/lth/bin N/A No Symlinks managed by rpkg pointing to active versions in /lth/store.
/cfg @cfg Yes System configuration. Snapshotted atomically with /lth/system before every rpkg update.
/user @home Yes Persistent user data. Not included in system rollbacks.
Ownership Model
System-Owned
/lth/systemkernel + lythd
/lth/storerpkg writes
/lth/binrpkg symlinks
User-Adjacent / Mutable
/cfgsnapshotted
/userpersistent
Snapshot Invariant
/lth/system and /cfg are always snapshotted and rolled back as an atomic pair. They can never be out of sync — a system update always carries its configuration context, and a rollback always restores both together.
§4 Tooling Stack

Tools are split into two tiers by function. The lyth- prefix denotes kernel-adjacent daemons. The rp- prefix denotes user-facing OS tools. The boundary is deliberate — a lyth- process being in the critical set vs. an rp- process being restartable is a design guarantee, not a convention.

lyth- Daemons

lythd PID 1, init, supervisor Yes Manages boot, stability timer, rollback coordination, service supervision loop
lythdist Capability distributor Yes Allocates unforgeable tokens at boot. Must complete before any other service can communicate
lythmsg IPC bus Yes Async shared-memory message passing. Crash triggers rollback. Transport and discovery both handled here

rpview TUI monitor Live service dependency graph and lythmsg bus health. Read-only — cannot affect system state
rpbreak Chaos engineering Manually kills services to test supervision, restart behavior, and rollback triggers under controlled conditions
Shell
ToolDescription
lysh Native system shell. POSIX-compliant. Launched as a user session process on demand. Not part of the boot sequence. Not in the critical set.
Service Definitions

lythd reads service definitions at boot. Each service is defined in TOML. The critical field marks a service as part of the rollback-triggering set.

[service]
name = "lynet"
bin  = "/lth/bin/lynet"
critical = false   // non-critical: restarted on crash, no rollback
deps = ["lythmsg"]

A service with critical = true is added to the rollback set. A failure in that service during the stability window triggers the same rollback path as a core daemon failure. By default, only lythd, lythdist, and lythmsg have critical = true.

Dependency Ordering
Services with unfulfilled deps are blocked until all listed dependencies are live. lythd resolves the dependency graph before starting any non-critical service. Circular dependencies are a boot error.
§5 Package Manager

rpkg is source-first. It always builds from Cargo. The local binary cache skips recompilation when the same content hash already exists in /lth/store. Btrfs reflinks deduplicate identical build outputs without copying.

Install Flow
StepAction
1Snapshot /lth/system and /cfg atomically as a pair
2Set rollback flag — activates 30-second stability timer on next boot
3Resolve dependencies via Cargo. For C deps without a -sys crate, process [system-deps] block first: build, install to /lth/store, inject paths into Cargo environment
4Check /lth/store for a matching content hash. If found, reflink — skip build
5If no cache hit: build via Cargo, install result to /lth/store
6Update /lth/bin symlinks atomically to point to new store entries
7On successful boot past stability window: mark snapshot stable, clear rollback flag
Dependency Resolution

rpkg resolves dependencies in two tiers, reached in order:

// Tier 1: Cargo-native
build.rs and -sys crates → Cargo handles natively

// Tier 2: system-deps block (C deps without -sys crate)
[system-deps]
libfoo = "2.1"

// rpkg: build libfoo → install to /lth/store
// inject PKG_CONFIG_PATH, LIBRARY_PATH → proceed with Cargo

Tier 2 only activates when a C dependency has no corresponding -sys crate. The package definition must declare a [system-deps] block; rpkg builds those dependencies first, installs them to /lth/store, and injects their paths into Cargo's environment (PKG_CONFIG_PATH, LIBRARY_PATH, etc.) before proceeding with the main build.

§6 Standard Library & Runtime

RaptorOS uses musl libc as its stable ABI between userspace and Lythos syscalls. musl is statically linked by default, which pairs naturally with rpkg's content-addressed store and reflink deduplication strategy.

PropertyValue
libcmusl — statically linked, POSIX-compliant
Rust stdFull std support via the x86_64-unknown-linux-musl target
Syscall interfacemusl acts as the stable ABI layer over Lythos kernel syscalls
LinkingStatic by default. rpkg reflinks deduplicate identical musl copies in /lth/store
Static Linking and the Store
Every binary in /lth/store is self-contained — no runtime linker resolution, no dependency on shared library paths. Btrfs reflinks mean that identical musl copies across packages occupy only one block on disk despite appearing as separate files.
§7 Resilience & Observability
Fault Isolation

Drivers and non-critical services run as isolated userspace processes. A crash in lynet, lygpu, or any non-critical service does not affect the kernel or critical daemons. lythd detects the crash via its supervision loop and restarts the service.

Isolation Guarantee
The Lythos capability model means a crashed userspace driver cannot corrupt kernel state — it holds only the capabilities it was granted, and those are revocable. A restart is always safe.
Automatic Rollback

rpkg snapshots /lth/system and /cfg as an atomic pair before every update and sets a rollback flag. On the first boot after an update, lythd starts a 30-second stability timer.

Trigger
Critical daemon (lythd, lythdist, or lythmsg) fails within 30-second stability window
Action
lythd triggers atomic Btrfs rollback of /lth/system and /cfg snapshots as a pair
Result
System reboots to last stable state. Config is always consistent with system — they are never out of sync
Config Invariant
Config always rolls back with the system. They are snapshotted as a pair and restored as a pair. There is no state where system and config are from different generations.
