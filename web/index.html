Core design

Everything in the kernel
is intentional

No monolithic bloat. Each subsystem is minimal, tested, and wired through capability handles your processes must earn.

01

Capability-Based Security

Every resource (memory, IPC endpoints, rollback authority) is governed by an unforgeable capability handle. No ambient authority. Least privilege by construction.

02

Preemptive Scheduler

Round-robin over a ready queue. Cooperative yields via SYS_YIELD. APIC timer-driven preemption. Blocking IPC with automatic task wakeup.

03

Ring-Buffer IPC

4 KiB shared pages per endpoint. Fixed 64-byte message slots. Blocking send/recv with task wakeup. Basis of the lythd boot protocol.

04

ELF64 Loader

Full ET_EXEC loader: PT_LOAD segments, user stack allocation with guard pages, BSS zero-fill, inherited capability tables, ring-3 via iretq.

05

4-Level Paging

NX, U/S, and huge-page support. Higher-half kernel window, identity-mapped lower GiB, per-process user stacks with guard pages and gap slots.

06

APIC + IOAPIC

Local APIC with calibrated timer and tick counter. IOAPIC routing replaces legacy 8259 PIC. Full RTE programming with masking per GSI.

About

About Lythos

Lythos is an experimental x86-64 microkernel developed by Pureshade Labs, LLC. Built from scratch in Rust, it explores what a capability-based security model looks like when taken seriously from the first line of code, not bolted on as an afterthought.

The design philosophy: every resource the kernel manages is accessed through an unforgeable capability handle. Memory, IPC endpoints, rollback authority. All gated. Processes hold only what they're explicitly granted, and no more. This is the principle of least privilege applied at the kernel level.

Lythos isn't trying to be Linux. It's trying to be a correct, minimal, verifiable foundation- the kind you'd actually trust to run security-sensitive workloads.

The project is open. We're committed to building in public and sharing the design decisions that shaped the kernel.

Lythos Microkernel Documentation

Internal documentation covering kernel internals for developers working on or extending Lythos. For writing userspace software that runs on Lythos, see the OROS repository.

Architecture

Lythos is a capability-based microkernel. The kernel provides memory management, scheduling, IPC, and a capability system. All device drivers, filesystems, and services run as userspace processes. Isolation is enforced through hardware paging and the capability system: a process can only access a resource if it holds a capability that grants it.

The kernel is single-processor. Multi-AP support requires a per-CPU scheduler and APIC startup sequence (tracked in TODO.md).

Critical invariants

BSS zeroing. boot.s zeros __bss_start...__bss_end before building page tables. MB1 header sets bss_end_addr = 0 to skip loader zeroing.

U/S bit propagation. Every page table level must have U/S set for user-mode accesses. vmm::walk_or_create propagates entry |= 1 << 2 on all intermediate tables when the User flag is in flags.

No map_page in 0–1 GiB. That range uses 2 MiB huge pages. Calling map_page there trips the PS=1 guard in walk_or_create.

Single-threaded allocators. alloc_frame and NEXT_STACK_SLOT are not locked. Safe while single-CPU with cooperative scheduling. A spinlock wrapper is required before SMP.

SpinLock disables interrupts. serial::SpinLock::lock saves RFLAGS and executes cli; guard restores via popfq on drop. Never call yield_task while holding a SpinLock.

Boot sequence

Lythos boots via Multiboot1 or Multiboot2. QEMU loads the kernel ELF at physical 0x100000 and jumps to the 32-bit entry in .boot. The stub switches to 64-bit mode and calls kmain.

Stage 1: 32-bit stub (boot.s)

1. Multiboot headers present (MB1 + MB2); GRUB/QEMU selects one.
   MB1 sets bss_end_addr = 0; stub owns BSS zeroing.
2. rep stosl clears __bss_start...__bss_end before any Rust runs.
3. 16 KiB temporary stack in .bss used during boot.
4. Minimal 4-level page table: identity-maps first 1 GiB with 2 MiB huge pages.
5. EFER.LME + CR0.PG → long mode. Far jump loads 64-bit CS.
6. Push Multiboot magic + info pointer, call kmain (never returns).

lythd ELF at boot

QEMU writes the lythd ELF to physical 0x400000 via -device loader,file=lythd,addr=0x400000,force-raw=on. PMM reserves frames 0x400000..0x480000 (512 KiB) so no allocator call overwrites the ELF. kmain reads from 0x400000 if the ELF magic is present, falling back to the embedded LYTHD_ELF byte slice.

Physical Memory Manager

Bitmap allocator in src/pmm.rs. One bit per 4 KiB frame; bit 0 = free, bit 1 = used. 4 GiB max = 1 M frames = 128 KiB bitmap stored in BITMAP: [u64; 16384] in .bss.

Init order

1. All frames marked used (bitmap = all-ones).
2. Parse Multiboot1/2 mmap; mark type=1 (available) regions free.
3. Re-mark kernel image frames used (KERNEL_START..KERNEL_END).
4. Re-mark frame 0 used (BIOS data / IVT).
5. Re-mark 0x400000..0x480000 used (lythd ELF blob).
6. Count free frames → store in FREE_FRAMES atomic.

API

init(mb_magic, mb_info)         One-shot init; call before any alloc
alloc_frame()                   Allocate one 4 KiB frame → Option<PhysAddr>
free_frame(addr)                Return frame to pool; panics on double-free
alloc_frames_contiguous(n)      Allocate n physically-contiguous frames
free_frames_contiguous(addr,n)  Free n contiguous frames
free_frame_count()              Current free frame count

Virtual Memory Manager

Standard x86_64 4-level paging in src/vmm.rs: PML4 → PDPT → PD → PT. Each level maps 9 bits. Leaf entries are 4 KiB (except the identity-map huge pages built by boot.s).

Page flags

Bit  Name   Meaning
───  ─────  ────────────────
0    P      Present
1    Write  Writable
2    User   Ring-3 accessible
63   NX     No-execute (XD)

Common combinations:
0x1  (P)      Read-only kernel page
0x3  (P+W)    Read-write kernel page
0x5  (P+U)    Read-only user page
0x7  (P+W+U)  Read-write user page
0x9  (P+NX)   No-exec kernel data

API

Do not call map_page on 0–1 GiB. Those are huge pages. walk_or_create panics with "huge page encountered" on a PS=1 PDE.

Tasks & Scheduler

Defined in src/task.rs. A task is the kernel's unit of execution. Every task has a unique TaskId (u64), a state, a 64 KiB kernel stack, a saved register context, a capability table, and a PML4 root. Task 0 is the bootstrap task (kmain).

Context switch

Software-only. switch saves callee-saved registers (RBX, R12–R15, RBP) + RSP for the outgoing task, restores them for the incoming task. The saved RIP is the return address pushed when switch was called.

User stack layout

Slots start at 0x0000_7FFF_0000_0000. Each slot = 2050 pages:

  [0]        guard page   (unmapped; fault on stack overflow)
  [1..2048]  usable stack (P + W + U + NX, 8 MiB)
  [2049]     gap page     (unmapped; separates slots)

LAPIC timer

Timer ISR (vector 32) fires at a calibrated rate measured against PIT in apic::init. Each tick increments TICK_COUNT atomic and calls yield_task; this is the preemption mechanism. Edge-triggered, locally-delivered, does not go through IOAPIC.

RSP0 / ring switching

On syscall or interrupt from ring 3, the CPU loads TSS.RSP0 as the kernel stack pointer. tss::init sets RSP0 to the top of the current task's kernel stack. RSP0 is updated on every context switch since only one task runs at a time.

Syscall reference

Entry via syscall instruction (LSTAR = syscall_entry). FMASK clears IF; interrupts disabled during handler. sysretq restores user RIP from RCX and RFLAGS from R11.

SYS_YIELD (0)

Voluntarily yield to the next ready task. Calling task placed at back of ready queue. Returns 0.

SYS_TASK_EXIT (1)

Terminate the calling task. Marks task Dead, switches to next. Frees kernel stack and capability table. Never returns.

SYS_MMAP (2)

Map a physical frame into the calling task's address space.

a1  virtual address   (4 KiB-aligned)
a2  physical address  (4 KiB-aligned)
a3  page flags        (bit 0=P, 1=W, 2=U, 63=NX)
→   0 on success, EINVAL if unaligned

Caller must hold a Memory capability. Currently validates cap presence; range restriction is planned.

SYS_MUNMAP (3)

Unmap a virtual page (a1 = virtual address, 4 KiB-aligned). Calls vmm::unmap_page + invlpg. Physical frame is not freed; caller tracks ownership.

SYS_CAP_GRANT (4)

a1  source handle in caller's table
a2  target task ID
a3  rights mask (result = source_rights & mask)
→   new handle in target's table, or ENOCAP / ENOPERM / EINVAL

Cannot grant to self. Target task must exist. Requires GRANT right.

SYS_CAP_REVOKE (5)

a1 = handle to revoke. Sets slot to None. Requires REVOKE right. Returns 0 or ENOCAP/ENOPERM.

SYS_IPC_SEND (6)

a1  IPC cap handle
a2  message buffer pointer (user space)
a3  message length in bytes (max 64)
→   0 on success; blocks if ring full

Requires WRITE right. Validates [a2, a2+a3) is in user address space before copying.

SYS_IPC_RECV (7)

a1  IPC cap handle
a2  destination buffer pointer (user space)
a3  buffer length in bytes
→   bytes copied (up to 64); blocks if ring empty

Requires READ right. Returns min(a3, MSG_SIZE).

SYS_IPC_CREATE (8)

No arguments. Allocates one 4 KiB physical page for a new ring buffer, maps it at 0xFFFF_D000_0000_0000 + idx * 4096 in the kernel IPC window. Returns new IPC capability handle with ALL rights.

SYS_ROLLBACK (9)

No arguments. Requires Rollback capability. Currently a stub that panics with "rollback invoked". Full checkpoint/restore is a future implementation. Returns ENOPERM if no Rollback cap.

SYS_EXEC (10)

a1  pointer to ELF data (caller's address space)
a2  ELF data length in bytes
a3  pointer to capability handle array (u64[])
a4  number of capabilities to inherit
→   new TaskId on success

Copies ELF + cap list from user space, parses, loads segments, allocates user stack, writes ABI frame, spawns exec trampoline. New task's cap table: handle 0 = caps[0], handle 1 = caps[1], etc.

SYS_BLK_READ (20)

a1  sector number (u64, 0-based)
a2  pointer to 512-byte destination buffer
→   0 on success; ENOSYS if no block device

SYS_BLK_WRITE (21)

a1  sector number (u64, 0-based)

Capability system

A capability is an unforgeable token granting a task the right to perform a specific operation on a specific resource. User code holds an opaque CapHandle(u64) and passes it to a syscall. The kernel looks up the handle in the task's CapabilityTable and checks the rights field.

Capability kinds

Kind      Resource            Grants access to
────────  ──────────────────  ─────────────────────────────────────────
Memory    Physical memory     SYS_MMAP: map arbitrary physical frames
Ipc       One IPC endpoint    SYS_IPC_SEND / SYS_IPC_RECV
Rollback  Kernel checkpoint   SYS_ROLLBACK

Rights bitfield

Bit  Constant    Meaning
───  ──────────  ────────────────────────────────────────────
0    READ  = 1   Can read / recv from the resource
1    WRITE = 2   Can write / send to the resource
2    GRANT = 4   Can grant a derived cap to another task
3    REVOKE= 8   Can revoke a cap from self or descendants
     ALL   = 15  All four rights combined

Rights can only be reduced in transit: grant result = source_rights & mask. No escalation path.

CapabilityTable

Each task has one CapabilityTable: a Vec<Option<Capability>> indexed by CapHandle. A None slot = revoked or never allocated. Handle numbers are assigned sequentially and never reused within a table, preventing use-after-revoke races.

Operations

cap_grant(from_table, handle, target_task_id, to_table, rights_mask)
  1. Lookup handle in from_table → ENOCAP if absent
  2. Check GRANT right → ENOPERM if missing
  3. new rights = source.rights & rights_mask
  4. Append Capability to to_table with reduced rights
  5. Return new handle index

cap_revoke(table, handle)
  Slot → None. Requires REVOKE right.

cap_cascade_revoke(table, handle, resolver)
  Revokes from holder + all derived holders (linear walk over live tasks).

Initial capabilities at boot

Handle  Kind      Rights  Description
──────  ────────  ──────  ──────────────────────────────────────────────
0       Memory    ALL     All free physical frames at boot
1       Rollback  ALL     Access to kernel rollback mechanism
2       Ipc       ALL     Boot IPC endpoint with pre-queued BootInfo

Security model notes

No ambient authority. A task with no capabilities cannot allocate memory, communicate, or spawn children.

Handle opacity. Sending a CapHandle value over IPC grants nothing; only SYS_CAP_GRANT transfers the actual capability.

Planned: range-restricted Memory caps. Currently grants access to all physical memory. A planned improvement adds a [base, limit) range field enforced in SYS_MMAP.

IPC

Each IPC endpoint is a 4 KiB physical page shared between the kernel and any tasks holding a capability to it. All messaging is in-kernel copy; user data is copied from sender's address space into the ring, then from the ring into the receiver's address space. No shared memory between tasks at the IPC layer.

Ring buffer layout

Offset  Size  Field
──────  ────  ──────────────────────────────────────────────
0       4     head (u32)  index of next slot to consume
4       4     tail (u32)  index of next slot to produce
8       4088  data        circular buffer of 64-byte message slots

MSG_SIZE      = 64 bytes
RING_CAPACITY = 63 messages   (floor(4088 / 64))

empty: head == tail
full:  (tail + 1) % RING_CAPACITY == head

Kernel mapping

Each endpoint is mapped at 0xFFFF_D000_0000_0000 + endpoint_index * 4096 for kernel access. User tasks access only via SYS_IPC_SEND / SYS_IPC_RECV.

Send semantics

1. Validate IPC cap + WRITE right.
2. Validate user pointer [msg_ptr, msg_ptr+msg_len) in user space.
3. If ring full → block task, yield. Resume when receiver drains.
4. Copy ≤MSG_SIZE bytes into data[tail * MSG_SIZE], advance tail.
5. If a task is blocked waiting to recv → wake it.

Recv semantics

1. Validate IPC cap + READ right.
2. If ring empty → block task, yield. Resume when sender writes.
3. Copy min(buf_len, MSG_SIZE) bytes from data[head * MSG_SIZE], advance head.
4. Return bytes copied.
5. If a task is blocked waiting to send → wake it.

BootInfo message (64 bytes)

Offset  Size  Field
──────  ────  ──────────────────────────────────────────────────────
0       8     signature   0xB007_1000_B007_1000
8       8     mem_bytes   free_frames × 4096
16      8     free_frames PMM free frame count at boot
24      12    vendor      CPUID leaf 0 vendor (EBX||EDX||ECX, 12 bytes)
36      28    _pad        zeroed

Limitations

Fixed 64-byte messages. Large data: share a Memory cap, exchange addresses over IPC. No multicast, no priority, no timeout.

ELF loader

Minimal ELF64 loader in src/elf.rs. Statically linked, position-dependent (ET_EXEC) x86_64 only. No shared libraries, dynamic linking, or relocations.

exec() steps

pub fn exec(elf_data: &[u8], caps: &[CapHandle]) → TaskId

1. Validate ELF header: magic \x7FELF, ELFCLASS64, ELFDATA2LSB,
   ET_EXEC, EM_X86_64. Panics on mismatch.

2. For each PT_LOAD segment:
   a. Allocate frames covering [p_vaddr, p_vaddr + p_memsz).
   b. Map frames at p_vaddr with flags from p_flags:
        PF_X(1) → no NX  |  PF_W(2) → Write  |  PF_R(4) → Present
        Always set User bit (ring-3 accessible).
   c. Copy p_filesz bytes from ELF at p_offset.
   d. Zero-fill p_memsz - p_filesz bytes (BSS padding).

3. alloc_user_stack → next slot from NEXT_STACK_SLOT:
     slot_base = 0x7FFF_0000_0000 + slot × (2050 × 4096)
     [0]       guard (unmapped)
     [1..2048] usable stack, P+W+U+NX
     [2049]    gap (unmapped)

4. Write initial SysV ABI frame below stack top:
     argc    (u64) = 0
     argv[0] (u64) = 0  (NULL)
     envp[0] (u64) = 0  (NULL)
   Align RSP down to 16 bytes.

5. Clone caps slice into new task's CapabilityTable:
     handle 0 = caps[0], handle 1 = caps[1], ...

6. Spawn kernel task running exec_trampoline(entry, stack_top).
   Trampoline calls enter_userspace → iretq to ring 3:
     CS/SS = user segments, RIP = e_entry, RSP = stack_top, IF set.

Known limitations

No ASLR. No relocation support. No dynamic linker. No per-process PML4; all tasks share the kernel mapping (planned future step). Segment overlap not checked.

lythd: init process

lythd is the first userspace process and parent of all other processes. The kernel starts exactly one lythd instance at boot. If lythd exits, the kernel panics. lythd is part of the OROS userspace repository; the compiled ELF is embedded in the kernel image for bootstrap.

Boot protocol

Step 1: kmain calls exec(LYTHD_ELF, &[mem_cap, rollback_cap, boot_cap])

  Handle 0  Memory    ALL  All free physical frames at boot
  Handle 1  Rollback  ALL  Kernel rollback / checkpoint access
  Handle 2  Ipc       ALL  Boot IPC endpoint with one pre-queued BootInfo

Step 2: lythd's first syscall:
  SYS_IPC_RECV(handle=2, buf, 64)   ← consume BootInfo

Step 3: lythd calls SYS_EXEC to launch lythdist (service manager),
  passing appropriate cap subset via caps array.

Step 4: lythd enters service loop:
  - Listens on IPC endpoint for service registration requests
  - Reaps dead tasks (no waitpid yet)
  - Holds Rollback cap; may invoke SYS_ROLLBACK on request

BootInfo message (64 bytes)

Offset  Size  Field        Description
──────  ────  ───────────  ────────────────────────────────────────────────
0       8     signature    0xB007_1000_B007_1000 (sanity check)
8       8     mem_bytes    free_frames × 4096 (usable RAM in bytes)
16      8     free_frames  PMM free frame count at the moment of boot
24      12    vendor       CPUID leaf 0: EBX||EDX||ECX (e.g. "GenuineIntel")
36      28    _pad         zeroed

Capability delegation policy

Memory. lythd should derive restricted Memory caps for each child (range restriction is a planned kernel feature; currently children receive a copy of the full Memory cap).

Rollback. Held by lythd only; should not be delegated to untrusted code.

IPC. lythd creates new endpoints via SYS_IPC_CREATE for each service and passes the capability to appropriate parties.

OROS relationship

The kernel specifies only the binary protocol: capabilities at fixed handles, BootInfo layout, syscall ABI. lythd's internal structure (lythdist, lysh, lythos-std) is an OROS concern.

IOAPIC (src/ioapic.rs)

Routes hardware interrupts from devices to local APICs. Replaces legacy 8259 PIC (remapped to vectors 32–47, then masked) with IOAPIC-based routing for VirtIO PCI interrupts.

MMIO interface

Offset   Register  Role
──────   ────────  ──────────────────────────────────────
+0x00    IOREGSEL  Write index of register to access
+0x10    IOWIN     Read/write the selected register

Index          Name      Contents
─────────────  ────────  ─────────────────────────────────
0x00           IOAPICID  IOAPIC ID in bits [27:24]
0x01           IOAPICVER Max redirection entry in [23:16]
0x10 + gsi*2   RTE_LO    Low 32 bits of redirection entry
0x11 + gsi*2   RTE_HI    High 32 bits (dest APIC ID [63:56])

RTE format (64 bits per GSI)

Bits   Field            Description
─────  ───────────────  ─────────────────────────────────────────
7:0    Vector           IDT vector the interrupt delivers to
10:8   Delivery mode    000 = Fixed
11     Dest mode        0 = Physical, 1 = Logical
12     Delivery status  R/O; 1 = pending
13     Pin polarity     0 = Active high
14     Remote IRR       R/O; level-triggered state
15     Trigger mode     0 = Edge, 1 = Level
16     Mask             1 = Masked (suppressed)
63:56  Destination      Target APIC ID

Init sequence + API

init()
  1. Map 0xFEC0_0000 → 0xFFFF_8000_FEC0_0000, write-through no-cache.
  2. Read IOAPICVER → MAX_ENTRY.
  3. Write all RTEs with IRQ_MASKED = 1 << 16.

map_irq(gsi, vector, flags)  Program one RTE; deliver to BSP (APIC ID 0)
mask_irq(gsi)                Set Mask bit
unmask_irq(gsi)              Clear Mask bit
entry_count()                Number of supported GSIs
set_phys_base(phys)          Override IOAPIC base (for ACPI MADT)

PCI scanner (src/pci.rs)

x86 Configuration Mechanism 1: write address to port 0xCF8, read/write data at 0xCFC in 32-bit chunks. 

CONFIG_ADDRESS format:
  Bit 31      Enable (must be 1)
  Bits 23:16  Bus number
  Bits 15:11  Device number (0–31)
  Bits 10:8   Function number (0–7)
  Bits 7:2    Register offset (dword-aligned)
  Bits 1:0    Always 0

find_device(vendor_id, device_id)
  Scans bus 0, devices 0–31, function 0 only.
  Returns PciDevice { bus, dev, vendor, device, io_bar0, irq_line }.
  Enables bus mastering (Command register bit 2) for DMA.

VirtIO block device (src/virtio_blk.rs)

VirtIO 0.9 (legacy) spec via PCI I/O BAR. PCI IDs: vendor 0x1AF4, device 0x1001. All I/O synchronous (polled). No interrupt-driven path yet.

Port register map (offsets from BAR0)

Offset  Size  Name              Direction
──────  ────  ────────────────  ─────────
0x00    4     DEVICE_FEATURES   R
0x04    4     GUEST_FEATURES    W
0x08    4     QUEUE_PFN         R/W
0x0C    2     QUEUE_NUM         R
0x0E    2     QUEUE_SEL         W
0x10    2     QUEUE_NOTIFY      W
0x12    1     DEVICE_STATUS     R/W
0x13    1     ISR_STATUS        R/W (write-to-clear)
0x14    4     BLK_CAPACITY_LO   R
0x18    4     BLK_CAPACITY_HI   R

Init sequence

DEVICE_STATUS = 0                               reset
DEVICE_STATUS = ACKNOWLEDGE (1)
DEVICE_STATUS = ACKNOWLEDGE | DRIVER (3)        negotiate features (none needed)
DEVICE_STATUS = ACKNOWLEDGE | DRIVER | DRIVER_OK (7)

Virtqueue layout (QUEUE_SIZE_MAX = 128, QUEUE_PAGES = 2)

Page 0:
  [0x000..0x800]  128 × 16-byte descriptors
  [0x800..0xA02]  Available ring: flags(u16) + idx(u16) + ring[128](u16)
Page 1:
  [0x000..0x408]  Used ring: flags(u16) + idx(u16) + ring[128](used_elem)
                  used_elem = id(u32) + len(u32)
QUEUE_PFN = vq_phys >> 12

3-descriptor I/O chain

Descriptor 0  request header (16 bytes, device-readable)
  type (u32): 0=read, 1=write
  reserved (u32): 0
  sector (u64): target sector

Descriptor 1  data buffer (512 bytes)
  Read:  VRING_DESC_F_WRITE (device-writable)
  Write: device-readable

Descriptor 2  status byte (1 byte, device-writable)
  0=success  1=error  2=unsupported

Submit flow:
  1. Fill descriptor chain.
  2. atomic::fence(SeqCst).
  3. Write 0 to QUEUE_NOTIFY (kick).
  4. Spin on used_ring.idx until it advances.
  5. Read + discard ISR_STATUS (clears spurious interrupt flag).

RFS: Raptor File System

Native filesystem for Lythos. Custom extent-based design for simplicity and correctness. No checksums (by design). Supports Unix permissions, symbolic links, 64-bit file sizes, and sparse files.

On-disk layout

Block  Contents
─────  ─────────────────────────
0      Superblock
1      Block bitmap
2–33   Inode table (32 blocks)
34+    Data blocks

Block size: 4096 bytes (8 × 512-byte sectors). All integers little-endian.

Superblock (block 0, first 64 bytes)

Offset  Size  Field         Value
──────  ────  ────────────  ──────────────────────
0       8     magic         RFS_V1\0\0
8       4     version       1
12      4     block_size    4096
16      4     total_blocks  image size / 4096
20      4     free_blocks   at format time
24      4     inode_count   1024
28      4     root_inode    0
32      4     bitmap_block  1
36      4     inode_start   2
40      4     inode_blocks  32
44      4     data_start    34
48      16    _pad          zeroed

Block bitmap (block 1)

One bit per block. Bit 0 of byte 0 = block 0. Set (1) = used. Blocks 0–33 always set. 4096 × 8 = 32768 blocks addressable = 128 MiB max.

Inode table (blocks 2–33)

32 blocks × 32 inodes per block = 1024 inodes. Each inode 128 bytes. Inode 0 = root directory.

Inode format (128 bytes)

Offset  Size  Field         Description
──────  ────  ────────────  ──────────────────────────────────────
0       4     flags         Inode status and type flags
4       2     mode          Unix permission bits (low 12 bits)
6       2     _pad0         Reserved, zeroed
8       4     uid           Owner user ID
12      4     gid           Owner group ID
16      4     nlink         Hard link count
20      8     size          File size in bytes (64-bit)
28      8     blocks        Number of 4096-byte blocks allocated
36      8     mtime         Modification time (Unix seconds, 64-bit)
44      8     ctime         Change time (Unix seconds, 64-bit)
52      4     ovfl_block    First overflow extent block (0 if none)
56      2     extent_count  Total number of extents
58      2     _pad1         Reserved, zeroed
60      64    extents[4]    Four inline extents (16 bytes each)
124     4     _pad2         Reserved, zeroed

Flags:
  Bit 0  INODE_USED     Allocated
  Bit 1  INODE_DIR      Directory
  Bit 2  INODE_SYMLINK  Symbolic link
  Bit 3  INODE_FAST_SYM Inline symlink target (≤64 bytes, stored in extents[])

Extent format (16 bytes)

Offset  Size  Field     Description
──────  ────  ────────  ─────────────────────────────────────────────
0       4     logical   First logical block in this extent
4       4     physical  First physical block number
8       4     count     Number of contiguous blocks
12      4     flags     Reserved, zeroed

Sparse: logical ranges with no covering extent → implicit zeros.
Lookup: scan extents for E where E.logical ≤ L < E.logical + E.count.
        physical = E.physical + (L − E.logical)

Overflow extent blocks

Offset  Size  Field        Description
──────  ────  ───────────  ───────────────────────────
0       4     next         Next overflow block (0=end)
4       4     used         Extents stored in this block
8       8     _pad         Reserved, zeroed
16      4080  extents[255] 255 × 16-byte extents

4 inline + 255 per overflow block = 259 extents with one overflow.
Chain grows arbitrarily.

Directory entry format

Offset  Size  Field      Description
──────  ────  ─────────  ────────────────────────────────────────
0       4     inode      Target inode number (u32)
4       2     rec_len    Total byte length of entry (u16)
6       1     name_len   Name length 1–255 (u8)
7       1     file_type  1=regular 2=directory 3=symlink (u8)
8       N     name       UTF-8, not null-terminated
8+N     P     padding    Zeroes to align rec_len to 4 bytes

- Last entry in a block: rec_len extended to fill remaining space.
- Entry with inode=0: deleted (hole). Scanning must check.
- Every directory has . and .. as its first two entries.

Fast symlinks

Symlinks with target ≤64 bytes are stored inline in inode.extents[] (bytes 60–123). INODE_FAST_SYM flag distinguishes inline from block-stored. Inline: read inode.size bytes from inode.extents as raw bytes. Block-stored: normal extent lookup.

mkrfs tool (tools/mkrfs/)

Host-side formatting tool for creating RFS disk images.

cd tools/mkrfs && make          # uses rustc directly (avoids kernel build-std)

./mkrfs <image> <size[K|M|G]> [<src-dir>]

./mkrfs disk.img 64M            # empty 64 MiB image
./mkrfs disk.img 64M rootfs/    # populate from rootfs/

Output:
  mkrfs: creating disk.img (67108864 bytes, 16384 blocks)
  mkrfs: done. 16384 total blocks, 16315 free, 42 inodes used.

Limits:
  Max image size:  128 MiB (32768-block bitmap)
  Max inodes:      1024 (fixed inode table size)
  No hardlinks     (each file gets a fresh inode)
  Timestamps from  host filesystem (stat.st_mtime)

Kernel RFS driver (planned: src/rfs.rs) 

1. Mount: read and validate the superblock.
2. Block I/O: wrap virtio_blk read/write into block-granular interface
   (8 sectors per block).
3. Inode read: compute block and offset from inode number.
4. Extent traversal: scan inline then overflow chain to resolve
   logical block → physical block.
5. File read: walk extents, copy data to caller buffer, fill sparse
   gaps with zeroes.
6. File write: allocate blocks, write data, update extents and inode.
7. Directory scan: iterate variable-length entries in directory blocks.
8. Path resolution: walk /-delimited components, resolve symlinks.
9. VFS interface: open, read, write, close, stat, readdir.

FAQ

Frequently asked questions

Monolithic kernels run drivers and filesystems in ring 0. A bug in a driver is a kernel bug: it can corrupt memory, escalate privileges, or crash the whole system. Lythos moves these components to userspace, where capability handles gate every resource access. A misbehaving driver can only damage what it was explicitly granted, and can be restarted without a reboot.
lythd is the first userspace process, analogous to PID 1 on Unix. The kernel starts it at boot with three capabilities (Memory, Rollback, and a boot IPC endpoint). It reads a BootInfo message, then spawns lythdist (the service manager) and enters a service loop. If lythd exits, the kernel panics; there is no recovery from a dead init.
No, and intentionally so. Lythos has its own minimal syscall ABI (11 syscalls at present). Linux compatibility is an optional OROS userspace concern (an OROS server could implement a Linux syscall translation layer), but it is not in the kernel and is not a current goal. The Lythos ABI is designed to be clean and correct, not compatible.
Lythos is developed by Pureshade Labs, LLC and is currently source-available for inspection and research. A formal open-source license will be announced alongside a stable release. Check the GitHub repository for the current license file.
Open an issue or pull request on the GitHub repository. Areas most welcoming of contributions: documentation, OROS userspace tooling, and test coverage. For kernel changes, open an issue first to discuss the design; the kernel ABI is intentionally stable and changes require careful review.
Lythos targets x86-64 (AMD64). It is tested under QEMU with a VirtIO block device, LAPIC, and IOAPIC. It should boot on real x86-64 hardware via any Multiboot1 or Multiboot2 compliant bootloader (e.g. GRUB), but real-hardware testing is ongoing. Multi-processor (SMP) support is tracked in TODO.md and not yet implemented.
Lythos is also the name of the broader protocol suite that the kernel anchors. The word is coined, a blend chosen for its feel rather than a direct translation. Internally it evokes lithos (stone, foundation) and a sense of deliberate, minimal construction. The kernel was previously codenamed CASK during early development.
